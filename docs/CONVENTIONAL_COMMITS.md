# Conventional Commits Review

## Conclusion

The repository does **not consistently follow** the [Conventional Commits](https://www.conventionalcommits.org/) specification.

## What was reviewed

The recent Git history was reviewed with:

```bash
git log --oneline --decorate -n 30
```

## Findings

Several commit messages use free-form summaries instead of the Conventional Commits format (`type(scope?): description`). Examples from the recent history include:

- `Remove dead versionConditions blocks and add NaN/unknown version test cases`
- `Create index.html`
- `Address review feedback: fix NaN handling, remove redundant conditionals, fix test path`
- `Initial plan`
- `Fix browser version compatibility checks`

There are also merge commits and automation/version bump commits such as:

- `Merge pull request #3 from ...`
- `Bump version to 26.0.24 [skip ci]`

While some of these messages are descriptive, they do not use the Conventional Commits structure.

## Recommendation

If the project wants to adopt Conventional Commits, future commit messages should follow this pattern:

```text
type(scope?): short description
```

The `scope` is optional. To mark a breaking change, append `!` before the colon:

```text
feat(api)!: remove deprecated endpoint
```

A `BREAKING CHANGE:` footer in the commit body can also be used to describe the change.

Common types:

- `feat`: a new feature
- `fix`: a bug fix
- `docs`: documentation-only changes
- `refactor`: code change that neither fixes a bug nor adds a feature
- `test`: adding or updating tests
- `chore`: maintenance tasks
- `build`: build system or dependency changes
- `ci`: CI/CD workflow changes

## Examples for this repository

- `fix(parser): handle NaN browser versions safely`
- `test(sicd): add unknown version coverage`
- `docs(readme): clarify automated version bump workflow`
- `chore(release): bump version to 26.0.25`

## Agent and assistant commit policy

Copilot-authored commits and Codex-authored commits should follow the **same Conventional Commits rules** as human-authored commits.

That means:

- Copilot branches, Codex branches, and direct human commits should all use `type(scope): description` for non-generated changes.
- Review-driven follow-up commits from Copilot or Codex should still use a proper Conventional Commit, such as `fix(docs): clarify agent commit requirements`.
- If Copilot or Codex creates documentation-only changes, use `docs(...)`.
- If Copilot or Codex fixes a bug, use `fix(...)`.
- If Copilot or Codex adds tests, use `test(...)`.

This keeps commit history consistent no matter whether the author is a person, Copilot, or Codex.

## Suggested adoption policy

1. Require Conventional Commits for all human-authored commits.
2. Require the same Conventional Commit format for all Copilot-authored and Codex-authored commits.
3. Prefer automated release/version bump commits under `chore(release): ...`. The current release workflow uses `Bump version to X [skip ci]`; treat this as an allowed exception until the bump script is updated to emit a Conventional Commit.
4. Keep merge commits as generated by the hosting platform, or use squash merges with a Conventional Commit title.
5. Optionally add commit linting in CI to enforce the format.

## Practical summary

- **Current state:** not consistently compliant.
- **Recommended state:** adopt Conventional Commits for all new non-generated commits from humans, Copilot, and Codex.
